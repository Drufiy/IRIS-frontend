Safety

System control should feel deliberate, not opaque.

IRIS is built around visible state, explicit approvals, and clear provider boundaries so the assistant can act without becoming silent or reckless.

Design Principles

Four things safety means in IRIS.

Classification

Actions are not treated equally

IRIS distinguishes between safe, warned, and dangerous actions so execution paths can stay proportional to actual risk.

Approval

Destructive steps stay explicit

Potentially dangerous actions go through a confirmation layer in the UI instead of silently happening in the background.

Visibility

State is always part of the experience

The overlay language makes it clear whether IRIS is idle, listening, acting, or waiting for a human decision.

Boundaries

Providers and memory stay explicit

Local memory, cloud APIs, and remote proxy mode are all configurable rather than hidden behind a vague black box.
