Experience Modes

Three layers of desktop presence.

The IRIS app should not behave like a single monolithic screen. It should adapt to the user's level of attention and the task in motion.

Ambient

Overlay when you only need signal

Keep IRIS light and peripheral when it is mostly listening, updating state, or waiting for the next command.

  • Idle, listening, acting, approval states
  • Low-clutter border and toast language
  • Fast voice-first feedback loop
Focused

Command center when work gets heavier

Expand into a full workspace when the user needs to inspect memory, actions, files, or multi-step execution flow.

  • Conversation and action timeline
  • Memory and reasoning surfaces
  • Provider settings and mode switching
Deliberate

Approval surfaces when risk appears

Escalate into explicit approval UI when the assistant is about to do something destructive or meaningfully consequential.

  • Clear request summary
  • Human confirm or deny action
  • Visible consequences before execution

Core Areas

What the desktop app needs to contain.

Conversation

Fast interaction panel

A running conversation surface that supports voice-first use but still works clearly when text, actions, and approvals need to be reviewed.

Actions

Execution history and approvals

A place to inspect what IRIS did, what it plans to do next, and where human approval is currently required.

Memory

Context and recall surfaces

Show what IRIS remembers, what it pulled into the current task, and where a user may want to prune or tune memory behavior.

Providers

Local and remote configuration

Expose provider modes, keys, and routing in a product surface so users do not have to live in raw config files forever.

Platforms

One desktop language, two operating systems

The UI system should be designed once and adapted cleanly for both Windows and macOS rather than split into separate product experiences.

Operating Loop

The desktop experience should support a full work cycle.

A strong IRIS desktop app is not just where responses appear. It is where requests become tracked work, where context accumulates, and where risky steps become explicit.

01

Capture intent fast

Voice first when speed matters, text when precision matters, and both reflected back into the same session.

02

Show what is happening

Users should be able to see active actions, execution history, and relevant context without opening logs or config files.

03

Hand control back cleanly

Approvals, cancellation, editing, and reruns should all feel native to the workflow instead of bolted on afterward.

Platform Parity

Shared core, native fit.

IRIS should feel like one product across platforms, but not like a rigid port that ignores how each desktop environment behaves.

Shared Across Both
  • State color system and motion language
  • Conversation, actions, memory, providers
  • Approval and safety interaction model
  • Voice-first workflow and execution timeline
Adapt Per Platform
  • Window chrome and native shell expectations
  • Platform-specific shortcuts and menu patterns
  • OS-tailored tray, dock, and background behavior
  • File system and automation edge cases

UI Principles

Rules worth carrying from web to desktop.

01

State-first color

Color should communicate listening, action, caution, and completion rather than acting as decoration.

02

Low-clutter surfaces

The UI should stay crisp and quiet, even when multiple systems like memory, approvals, and tasks are visible together.

03

Visible consequences

The app should always make it obvious what IRIS is about to do, what it already did, and when it is waiting on the user.
